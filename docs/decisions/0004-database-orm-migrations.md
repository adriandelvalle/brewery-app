# ADR-0004: Database, ORM & Migrations Strategy

---

## Context

`brewery-app` needs a relational database to persist inventory, batches, and fermentation data.
Decisions required:

1. Which database engine for development and production.
2. Which ORM / query layer.
3. How to manage schema changes safely over time.
4. How to handle credentials before HashiCorp Vault is available (Phase 2, Week 7).

---

## Options Evaluated

### Database engine

| Option     | Verdict  | Reason                                                                   |
| ---------- | -------- | ------------------------------------------------------------------------ |
| SQLite     | Rejected | No concurrent writes, no row-level locks. Not representative of prod.    |
| PostgreSQL | Accepted | Industry standard, native JSON support, full ACID, runs well in Docker.  |
| MySQL      | Rejected | Weaker JSON support, licensing ambiguity in some cloud providers.        |

### ORM

| Option        | Verdict  | Reason                                                                    |
| ------------- | -------- | ------------------------------------------------------------------------- |
| SQLAlchemy 2  | Accepted | De-facto Python standard, integrates with FastAPI via `AsyncSession`.     |
| Tortoise ORM  | Rejected | Async-native but smaller ecosystem, fewer MLOps integrations.             |
| Raw SQL       | Rejected | Valid at scale, not appropriate for a learning-path project.              |

### Migrations

| Option     | Verdict  | Reason                                                                          |
| ---------- | -------- | ------------------------------------------------------------------------------- |
| Alembic    | Accepted | Standard companion to SQLAlchemy, autogenerate support, reversible migrations.  |
| Manual SQL | Rejected | Error-prone, unversioned, incompatible with CI/CD pipelines.                    |

---

## Decision

- **Database**: PostgreSQL 16 (Docker container from Week 4 onward; local socket for Week 3 dev)
- **ORM**: SQLAlchemy 2.x with async driver (`asyncpg`)
- **Migrations**: Alembic from day one — no schema changes without a migration file
- **Secrets (pre-Vault)**: `python-dotenv` + `.env` file (600 permissions, gitignored) + `.env.example` committed to repo

### Pre-Vault secrets pattern (Weeks 3–6)

This is intentional technical debt, accepted and documented.

```
# .env (gitignored, permissions 600)
DATABASE_URL=postgresql+asyncpg://brewery:secret@localhost:5432/brewery_db

# .env.example (committed)
DATABASE_URL=postgresql+asyncpg://USER:PASSWORD@HOST:PORT/DB_NAME
```

Migration to Vault (Week 7) will replace `DATABASE_URL` with dynamic credentials
via Vault's PostgreSQL secrets engine. No application code changes required —
only the source of the env variable changes.

### Alembic conventions

- One migration per schema change — never accumulate multiple changes in one file.
- Migration filenames: autogenerated by Alembic (`alembic revision --autogenerate`).
- Always review autogenerated migrations before applying — Alembic misses some changes
  (e.g. index renames, check constraints).
- Downgrade scripts are mandatory for all migrations in this phase.

### Project layout

```
backend/
├── src/
│   ├── db/
│   │   ├── base.py          # DeclarativeBase
│   │   ├── session.py       # AsyncSessionLocal, get_db dependency

```

## Consequences

| Type | Detail |
| ---- | ------ |
| ✅ | Schema changes are versioned, reversible, and reviewable in Git |
| ✅ | PostgreSQL in dev = PostgreSQL in prod — no environment mismatch |
| ✅ | AsyncSession integrates cleanly with FastAPI's dependency injection |
| ✅ | `.env.example` documents required variables without exposing secrets |
| ⚠️ | Alembic autogenerate requires models to be imported in `env.py` — easy to forget when adding new models |
| ⚠️ | `asyncpg` requires PostgreSQL running locally for Week 3 dev; mitigated by Docker Compose in Week 4 |
| ⚠️ | Pre-Vault credentials in `.env` are plaintext on disk — mitigated by 600 permissions and gitignore |

---

## Migration path to Vault (Week 7)

1. Enable Vault PostgreSQL secrets engine.
2. Replace static `DATABASE_URL` in `.env` with a Vault-issued dynamic credential.
3. Add Vault agent sidecar or `hvac` Python client to refresh credentials on expiry.
4. Mark this ADR as **Partially Superseded** by ADR-0007 (Vault integration) for the secrets section.
